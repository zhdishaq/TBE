# Quick Task 260424-20h: Share infra across services — SUMMARY

## Outcome

All 9 .NET API services + the YARP gateway now share a consistent external-infra contract:

- **Keycloak** — `https://openid.3ha.one` (external). Realms: `tbe-b2c` for customer/agent API, `tbe-b2b` for B2B admin service-accounts, `tbe-backoffice` for Crm/Backoffice services.
- **YARP gateway public URL** — `https://booking.3ha.one`. Services reference it as `Services:Gateway:BaseUrl`; intra-docker calls go through `Services:Gateway:InternalBaseUrl = http://tbe-gateway:8080`.
- **Internal service names unchanged** — `booking-service:8080`, `payment-service:8080`, etc. The gateway's `ReverseProxy.Clusters` still resolves them via docker DNS; only the external Keycloak + public gateway URL changed.

## Decisions

- **External Keycloak is authoritative** — removed the embedded `keycloak` service block from docker-compose; the leading comment already stated `openid.3ha.one` was the canonical endpoint, but the service was still defined. Made the comment match reality.
- **Dual Gateway URLs** — `Services:Gateway:BaseUrl` (public, for issuing tokens / redirect URIs) vs `Services:Gateway:InternalBaseUrl` (container DNS, for internal RPC). Services that need to round-trip through YARP go through the internal URL; anything embedded in emails / PDFs / OIDC redirects goes through the public URL.
- **Realm → service mapping preserved** — B2C services keep `tbe-b2c` as primary realm and `audience=tbe-b2c-api`. CrmService + BackofficeService switch to `tbe-backoffice` (they were previously missing Keycloak blocks entirely, so this just makes the defaults explicit).
- **Per-service KeycloakB2B blocks duplicated** — Plan 05-04 decision kept: BookingService and PaymentService each hold their own `KeycloakB2B` service-account config (different ClientIds: `booking-service` vs `payment-service`). Not consolidated because their `AgentAdminRole` defaults differ.
- **Credentials in `.env.example` scrubbed** — The prior `.env.example` shipped real-looking Stripe test keys, Sabre creds, Hotelbeds keys, and TBE_*_GATEWAY_CLIENT_SECRET values. Those are now empty placeholders. Real dev values live in the local (gitignored) `.env`.

## Pre-deploy checklist

1. Populate local `.env` from `.env.example` — at minimum set `KEYCLOAK_PUBLIC_URL=https://openid.3ha.one`, `GATEWAY_PUBLIC_URL=https://booking.3ha.one`, and the B2B/B2C/backoffice gateway client secrets.
2. Create a `booking-service` client in the `tbe-b2b` realm with service-account + view-users/query-users roles; populate `KEYCLOAKB2B__BOOKING__CLIENTSECRET`.
3. Ensure the reverse-proxy/ingress in front of the tbe-gateway container terminates TLS for `booking.3ha.one` and forwards the `X-Forwarded-*` headers YARP expects.
4. Confirm `openid.3ha.one` already serves the three realms (`tbe-b2c`, `tbe-b2b`, `tbe-backoffice`) with the audience `tbe-b2c-api` / `tbe-backoffice-api` configured on the respective clients.
