- **44px minimum touch target** on all interactive controls (inherited from Phase 5).

Exceptions: `h-9` audit timeline row height (documented above) is the only deviation from `h-11` compact density; every other table uses `h-11`.

## Typography

Sizes locked at 4 roles (identical to Phase 5 / Phase 4). Heading line-height 1.2 for display, 1.3 for heading; body line-height 1.5; label line-height 1.4.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 600 (semibold) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.3 |
| Display | 28px | 600 (semibold) | 1.2 |

Rules:
- Only **two weights** site-wide: 400 (regular) and 600 (semibold). No 500, 700, 800 usage.
- **All money amounts use `font-variant-numeric: tabular-nums`** — mandatory on MIS reports, reconciliation queue, wallet/commission summaries, markup rule values, manual wallet credit amounts, agency credit-limit fields, Agency Monthly Statement preview. Checker-validated.
- **All timestamps in audit/event tables use `tabular-nums` + ISO 8601** (`YYYY-MM-DD HH:mm:ss UTC`) — ensures columnar alignment in long event lists and unambiguous timezone handling (BookingEvents, DLQ, StripeEvents, communication log).
- **All IDs (GUIDs, PNRs, EventIds) render in monospace 12px via `font-mono text-xs`** — supports copy-paste into support tickets and GDS terminals without OCR-style ambiguity.
- Prices, balances, and credit limits render at body size (16px) + semibold, never display size (display reserved for page-level headings and the Agency Monthly Statement total).
- 12px/400/1.4 reserved for disclaimer/fine print (audit metadata sublines, tooltip body, MIS report generation notes, GDPR tombstone "this customer was erased on …").
- Role-gated UI affordances (mutating buttons visible to `ops-admin`/`ops-cs`/`ops-finance` but hidden for `ops-read`) use a **`data-role-gated="true"` attribute** so the UI auditor can grep-verify which controls expand/collapse per role. **No colored labels** on role-gated controls — absence + presence is the signal, not chrome.

## Color

Base palette inherited from Phase 5 (zinc scale + white + red-600 destructive + amber-600 warning + green-600 success + green-700 domain-specific commission text). Phase 6 swaps the accent only.

### Portal Differentiation Strategy

**Decision (CONTEXT D-47 — LOCKED):** Distinct accent color for Backoffice (`slate-900` on light / `slate-200` on dark) **+** a persistent "BACKOFFICE" wordmark badge in the header shell. Both-together carried forward from Phase 5 rationale:
1. Accent alone can be missed by colorblind users and on grayscale screenshots.
2. Badge alone is easy to glance past once familiarity builds.
3. Together they are redundant — a safety property (if a cookie ever leaks and renders a B2C or B2B page inside backoffice-web, the mismatch is instantly visible).

Slate-900 chosen over other hues because:
- **Operational gravitas** — backoffice actions (cancel, refund, erase, approve payout) are higher-stakes than B2C browsing or B2B selling. A neutral, near-black accent signals "this is the control room", not "this is a shopping experience".
- **Visually distinct** from B2C `blue-600` and B2B `indigo-600` at a glance (monochrome dark vs saturated blues) — zero Δ-hue ambiguity.
- **Passes WCAG AAA** on light background (`slate-900` on `white` ≥ 16.7:1) and WCAG AA on dark (`slate-200` on `zinc-950` ≥ 11.8:1).
- **Does not fight destructive red, warning amber, or success green** — slate is chromatically separated from all three status tokens.
- **Already present in Tailwind v4 default palette** — no new palette definition needed.

### Accent (`slate-900`) reserved-for list

Accent is applied **only** to the following elements. Any other use is a checker violation.

1. **Primary CTA button fill** — `Search bookings`, `Cancel booking`, `Create manual booking`, `Open wallet credit request`, `Approve wallet credit`, `Approve payout`, `Export to Excel`, `Save markup rule`, `Erase customer data`, `Resolve manually`, `Requeue`, `Save contract`, `Add note`, `Save threshold`.
2. **Selected-row left border on tables** — 4px `slate-900` left border + 1px ring on the focused row in unified booking list, CRM agency list, DLQ, reconciliation queue, MIS details drill-down, markup rule list.
3. **Active nav item on sidebar** — left 3px `slate-900` bar + `slate-900` icon + `slate-900` label; inactive items use `--foreground` icon + label with no bar.
4. **Focus ring** on keyboard-focused interactive elements (`--ring` overridden to `slate-700` in `backoffice-web/styles/globals.css` delta; light `slate-400` default stays for pre-interaction inputs).
5. **"BACKOFFICE" wordmark badge** in the header shell — outline style, 1px `slate-900` border + `slate-900` text on `--background`, never filled. Component: `<BackofficePortalBadge />`.
6. **Tab underline** on tab-grouped detail pages (CRM 360 profile/bookings/wallet/communications tabs; Booking Detail itinerary/events/payments tabs) — 2px `slate-900` underline on active tab.
7. **Data-grid column header on sort-active state** — sorted column shows `slate-900` text + sort arrow; unsorted headers use `--muted-foreground`.

**Not allowed for accent:** body links (use `--foreground` + underline on hover), headings, card backgrounds, decorative icons, status dots in lists (use green/amber/red semantic tokens), MIS chart fills (use `zinc-600` for primary series, `zinc-400` for secondary — charts never use accent). Commission-text column in MIS/statement preview (must be `green-700` — convention from Phase 5).

### Destructive (`red-600`) reserved-for list

1. **Staff-cancel booking button** in booking detail Actions panel (`ops-cs`/`ops-admin` — see 4-eyes rule).
2. **GDPR erase button** on `/backoffice/customers/{id}` detail page (`ops-admin` only; opens confirmation dialog per D-57).
3. **Resolve DLQ "manual failure"** button — mark a dead-letter as abandoned with a loss reason (`ops-admin` only).
4. **Form validation error text and field border** (inherited).
5. **"Cancelled" / "Failed" / "Erased" row status badges** in unified booking list and customer list.
6. **Reconciliation queue row chevron** for high-severity drift rows (ledger-vs-Stripe amount mismatch > £5) — row gets `red-50` / `red-950/40` background tint + `red-600` chevron.
7. **Deactivate agency** button on `/backoffice/agencies/{id}` (soft-disable — active agents cannot log in until re-enabled).

**Not allowed for destructive:** generic warnings (use `amber-600`), 4-eyes pending approvals (use `amber-600`), orphan-stripe-event rows below drift threshold (use `amber-600`).

### Warning (`amber-600`) reserved-for list

1. **4-eyes Pending Approval badge** — appears on rows/cards where a request awaits counter-signature (`BackofficeWalletCreditRequest` status=`PendingApproval`, `BookingCancellationRequest` status=`PendingApproval`). Chip: 1px amber border, amber-50 fill light / amber-950/40 dark, `lucide:clock-3` icon 12px, text `Awaiting ops-admin approval` (or `Awaiting approval`).
2. **DLQ unresolved row indicator** — chevron + row left-border `amber-600` on rows with `ResolvedAt IS NULL AND RequeueCount > 0` (churning retries) or fresh rows.
3. **Reconciliation low-severity drift** — orphan Stripe event + orphan wallet row within £5 → amber chip "Below review threshold".
4. **Agency credit-limit warning** on CRM 360 agency card when `currentBalance < 0 AND |currentBalance| > creditLimit × 0.8` (nearing overdraft ceiling).
5. **Low-balance tile** on agency CRM card — "Wallet low: £{balance} below £{threshold}".
6. **MIS anomaly flag** (drilled down in v2 but the token position is reserved now) — amber dot on rows where nightly aggregate diffs from yesterday's figure by > 50%.

### Success (`green-600`) reserved-for list

1. **Approved-by-ops-admin chip** — replaces the amber Pending Approval badge once ops-admin approves a wallet credit / payout / cancellation. Copy: `Approved by {actor}`.
2. **Resolved DLQ row dot** — green dot + "Resolved" status label once `ResolvedAt IS NOT NULL`.
3. **Reconciliation no-drift row dot** — when the nightly diff reconciles cleanly (Stripe event matched to a wallet row within tolerance).
4. **Booking "Confirmed" / "Ticketed" dot** in unified booking list (same as Phase 5 booking list).
5. **GDPR erasure complete toast** — sonner `type: "success"` after successful erasure.
6. **Supplier contract "Active" chip** (within validity date window).
7. **Sitewide toast for saved markup rule / saved threshold / added communication note** (non-blocking successes).

## Copywriting Contract

Voice: **direct, operational, neutral third person for customer/agency references ("the customer", "this agency"), second person for staff actions ("you", "your role").** British English spelling (`customise`, `organisation`, `whilst`). **Tone matches Phase 5 (formal and actionable)**. No exclamation marks. No emoji. Sentence case for all labels, titles, and buttons. Never "Submit".

Rationale: backoffice copy lives on the operational side of the platform. Every message is a potential audit-log line — it must name the thing that was done, not editorialise about it. "Customer data erased" not "Sorry — we've removed that customer's data". "4 bookings cancelled" not "4 bookings successfully cancelled!".

Dialog title names the action in the imperative. **Reason text is required** for every destructive confirmation in Phase 6 (write-to-audit-log control per D-48) — the confirm button stays disabled until the reason textarea has non-whitespace content. **Typed confirmation is required only for GDPR erasure** (D-57: the operator must type the customer's email address before the confirm button enables — belt-and-braces against fat-finger).

## Screen-by-Screen Contract

### 1. Login (unauthenticated)

### 2. Header Shell + Sidebar (every authenticated route)

- **Purpose:** persistent navigation + role-context identity.
- **Layout (desktop ≥ `lg`):**
  - Top bar `h-14`: brand wordmark `TBE` + `<BackofficePortalBadge />` outline pill (left); global search input (center, `cmdk`-powered); user menu + role chip + notifications bell (right, fixed right cluster).
  - Left sidebar `w-64`: nav tree grouped into 7 sections (see below). Each item is a `<Link>` with `lucide` icon 18px + 14px semibold label + optional count badge (e.g., DLQ unresolved count). Active item uses slate-900 left bar + slate-900 icon + label.
- **Layout (mobile < `lg`):** top bar `h-16`; sidebar collapses into a `lucide:menu` hamburger that opens a `sheet` from the left with the same nav tree. Global search collapses into a `lucide:search` icon that opens a full-screen overlay.
- **Sidebar sections (role-gated visibility):**
  1. **Overview** — Dashboard (all roles)
  2. **Bookings** — Unified booking list (all roles), Manual booking entry (ops-cs + ops-admin), Cancellation requests queue (ops-admin — badge count of pending)
  3. **Agencies** — Agency list (all roles), Markup rules (ops-finance + ops-admin), Commission payouts (ops-finance + ops-admin)

  5. **Finance** — Wallet credit requests (ops-finance + ops-admin — badge count of pending), Payment reconciliation (ops-finance + ops-admin), Supplier contracts (ops-finance + ops-admin)

- **User menu (top right):** name + email + role chip (one of `ops-admin` / `ops-cs` / `ops-finance` / `ops-read` — colored chip with `zinc-200` bg + `--foreground` text; role name is the signal, not the color). Dropdown: `My account`, `Sign out`.

- **Interaction:** global search debounced 300ms; results panel groups hits by entity type (Bookings / Customers / Agencies / PNRs) with top-5 preview rows and a "See all results" link.
- **Accessibility:** sidebar uses `<nav aria-label="Primary">`. "BACKOFFICE" badge has `aria-label="Backoffice portal"`. Role chip has `aria-label="Your role: {role}"`. Notifications bell has `aria-label="{N} unread notifications"` (0 when empty; still announced).

### 3. Dashboard (`/backoffice`)

- **Purpose:** single-pane overview of pending work across the four role areas.
- **Role gate:** all roles (content varies by role — `ops-read` sees counts but no action links).
- **Layout:** 3-column grid at `xl`+; 2-column at `lg`; single-column below.
  - **Tile 1 — Pending approvals** (ops-admin only; hidden for others): count of cancellation requests + wallet credit requests + draft commission payouts. Click through to queues.
  - **Tile 2 — Dead-letter queue** (ops-admin only; hidden for others): unresolved count + oldest-unresolved age. Amber if > 0.
  - **Tile 3 — Reconciliation** (ops-finance + ops-admin): drift items open, split by severity (high red / low amber).

## Interaction & Layout Contracts (phase-specific)

These are Phase 6 mechanics the planner and executor must follow verbatim. The checker validates against them.

### Portal Differentiation

- **`<BackofficePortalBadge />`** renders in the header shell on every authenticated route. 1px `slate-900` border (`slate-200` on dark), `slate-900` text (`slate-200` on dark), `--background` fill, 12px semibold, uppercase letters, tracking-wide. Never filled; never below 32px total height. `aria-label="Backoffice portal"`.
- **Primary accent token** is `slate-900` (light) / `slate-200` text / `slate-700` fills (dark), replacing Phase 5's `indigo-600`. This is the **only** Tailwind palette swap between b2b-web and backoffice-web.

### 4-Eyes Approval UX (D-48)

- **Only two action classes enforce 4-eyes:** (a) post-ticket manual wallet credit (D-53), (b) booking cancellation (per D-48).
- **Opening operator flow:** opens a request via the primary CTA on the source surface → confirmation dialog with required reason → request persisted with `Status=PendingApproval` + `RequestedBy` + `RequestedAt` + `ExpiresAt = RequestedAt + 72h` → amber pending-approval banner + amber chip on every list view the request surfaces on.
- **Approver flow:** ops-admin opens the request → sees the amber banner with requester info + original reason → enters approval reason in a required textarea → clicks `Approve` or `Deny` → status transitions.
- **Self-approval blocked:** `(RequestedBy == CurrentActor && CurrentRole == ops-admin)` → server returns 403 + the specific error from the Error states table; UI never hides the approve button on self-requests (so operators see why action is blocked).
- **Expiry:** request with `ExpiresAt < UtcNow AND Status=PendingApproval` → rendered with `status=Expired` red chip; cannot be approved or denied; only new request can be opened.
- **Non-4-eyes mutations (GDPR erasure, markup CRUD, cancellation by ops-admin acting on own request):** single-operator, attributed to the actor via audit-log entry. UI uses the **same Destructive-confirmations pattern** (reason required) for visual consistency, but no pending-approval badge.

### Role-Gated Controls

- Every mutating button uses `<RoleGatedButton />` wrapper. When the current session role is not in the button's allowed-roles list, the button renders **disabled** with `data-role-gated="true"` + tooltip "Your role ({role}) cannot perform this action."
- **Never hidden** — disabled + tooltipped is the rule. Discoverability matters for staff who may need to escalate requests. Exception: *destructive* buttons (cancel/erase/deactivate) can be **hidden** for `ops-read` to reduce visual clutter, since there's no pending-approval equivalent for ops-read to escalate from.
- Sidebar nav items follow the same rule for their primary-action child buttons (e.g., "Create manual booking" is disabled in the sidebar for ops-finance + ops-read).

### Audit-Log-Is-Primary Control (D-48)

- Every single-operator mutation writes an audit-log row with `Actor`, `ActorRole`, `Action`, `BeforeState`, `AfterState`, `Reason`, `OccurredAt`. Mutations without a reason cannot be submitted (client-side + server-side validation).
- For BookingEvents the audit entry IS the `BookingEvents` row itself (D-49/D-50). For mutations without a saga backing (markup, supplier contracts, GDPR erasure, DLQ resolve, reconciliation resolve, staff enable/disable), a dedicated audit-log table per service records the entry. Each table surfaces in the Agency 360 Audit tab / Customer 360 Audit tab / Markup audit log tab / DLQ resolved tab / Reconciliation resolved filter / Staff audit log.

### Global Search

- Shortcut: `Cmd/Ctrl+K` opens; `Esc` closes.
- Results scoped to the user's role (ops-read sees everything read-only; ops-cs doesn't see agency-only routes; etc.).
- Server debounce 300ms; cancel in-flight requests on new keystroke.
- Fan-out via `GET /api/backoffice/search?q=`; backend does parallel `Task.WhenAll` across Booking/Customer/Agency/PNR indices.

### Data-Grid Column Conventions

- All data-grids use the starterKit `data-grid-table` + `data-grid-pagination` primitives (same as Phase 5).
- All IDs + references render in `font-mono text-xs`. Clicking copies to clipboard + shows a sonner toast "Copied {ref}".
- All timestamps render as `YYYY-MM-DD HH:mm:ss UTC` with tabular-nums; hover shows relative time ("3 hours ago") via `<time dateTime={iso}>`.
- All money renders as `£{amount}` with tabular-nums, 2 decimal places, signed where applicable (credits green-700, debits default foreground).
- Sortable columns have a click-to-sort header with `lucide:arrow-up-down` icon default, `lucide:arrow-up` / `lucide:arrow-down` on active.
- Bulk actions (where supported — commission payout bulk approve) use a `Checkbox` in a dedicated leftmost column + a floating `Selection` toolbar at the top of the table.

### Responsive Breakpoints

Follow Tailwind v4 defaults (same as Phase 5): `sm` 640 / `md` 768 / `lg` 1024 / `xl` 1280 / `2xl` 1536. Desktop-first for backoffice (see "Mobile density" above).
- Sidebar: visible at `lg`+, hamburger below.
- Dashboard: 3-column at `xl`+, 2-column at `lg`, single-column below.
- All data-grids: table at `md`+, stacked row-cards below (with filter panel collapsible into a drawer).

### Forms, Loading, and Feedback

Inherited from Phase 5 verbatim:
- `react-hook-form` + `zod`. Validation errors below each field in `red-600` 14px; red 1px field border.
- Skeleton rows matching final card/row shape on list pages.
- `sonner` toasts for non-blocking successes only. Errors that require user action use inline alerts.
- Focus rings using `--ring` — Phase 6 override: `--ring` set to `slate-700` (light) / `slate-400` (dark) so focus outlines match the accent.

### Accessibility Baseline (WCAG 2.1 AA)

- All interactive elements keyboard-reachable; tab order matches visual order.
- All form fields have explicit `<label>` (not placeholder-only).
- Dialogs use Radix `Dialog` / `AlertDialog` — focus trapped, ESC closes, focus returns to trigger.
- All text meets 4.5:1 contrast on its background; large text (18.66px+) meets 3:1. Slate-900 on white meets AAA (16.7:1).
- **Color is never the only channel** — all status dots paired with labels; 4-eyes pending chips have icon + text; DLQ/reconciliation severity has icon + label; charts have "View as table" fallback.
- **"BACKOFFICE" badge** announced via `aria-label="Backoffice portal"`.
- Role chip in header announced as `"Your role: {role}"`.
- Pending-approval banner uses `role="status"` (informational). Destructive confirmation dialog titles use `aria-labelledby` targeting the dialog heading.
- Live-updating age counters in Cancellation / Wallet credit / DLQ queues use `aria-live="off"` on tick updates; `aria-live="polite"` only on threshold transitions (crossing from "< 24h" to "> 24h aging" style thresholds).
- Charts have `role="img"` + `aria-label` summarising the series; toggleable table fallback.
- Typed-confirmation (GDPR erasure) field has `aria-describedby` pointing to the "Type the customer's email to confirm" instruction.
