<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Gateway -> tbe-b2b JWT validation | Audience check enforces token was minted for this API; realm isolation from tbe-b2c |
| Next.js route handler -> Keycloak Admin API | Service-account token controls who can create/deactivate users |
| Client form body -> route handler | `agency_id` MUST come from session, never body (Pitfall 28) |
| Admin nav rendering based on session roles | Server-side check; client-side check only a UX hint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01-01 | Spoofing | Token audience confusion (Pitfall 4 / Pitfall 19) -- tbe-b2c token accepted on B2B endpoint | mitigate | Gateway flips `ValidateAudience=true` AND adds `.AddAuthenticationSchemes("tbe-b2b")` on both B2BPolicy + B2BAdminPolicy -- only tokens from the tbe-b2b realm reach B2B endpoints; cross-realm test case asserts 401 |
| T-05-01-02 | Elevation of Privilege | Non-admin agent calling POST /api/agents | mitigate | `/api/agents` route handler starts `if (!session?.roles?.includes('agent-admin')) return 403`; B2BAdminPolicy on downstream path; middleware bounces non-admin off /admin/*; grep-verifiable in 3 places for defense-in-depth |
| T-05-01-03 | Tampering | Client forges agency_id in request body (Pitfall 28) | mitigate | Route handler destructures body as `{ email, firstName, lastName, role }` only -- never reads body.agency_id; `createSubAgent(agencyId: session.user.agency_id, ...)`; unit test posts forged body `{ agency_id: "OTHER-AGENCY", ...}` and asserts the created user's agency_id equals the CALLER's, not the body's |
| T-05-01-04 | Information Disclosure | PII leak in Keycloak error response | mitigate | `createSubAgent` catches non-ok response WITHOUT echoing body (Keycloak reflects email on duplicate); log `response.status` only; return generic 502 to client |
| T-05-01-05 | Information Disclosure | Cross-tenant deactivation -- admin A deactivates admin B's user | mitigate | `deactivateUser` pre-fetches target user via Admin API and asserts `user.attributes.agency_id[0] === callerAgencyId` before PUT; throws if mismatch; logs `WARN cross-tenant deactivation attempt caller={callerAgencyId} target_owner={...} user={userId}` |
| T-05-01-06 | Elevation of Privilege | agent-admin creates another agent-admin (privilege escalation within agency) | mitigate | `createSubAgent` param type restricts `role: 'agent' \| 'agent-readonly'` -- `agent-admin` literally cannot be passed; TypeScript + runtime validation at route handler; route handler body schema rejects `role === 'agent-admin'` |
| T-05-01-07 | Denial of Service | Admin API token cache thundering herd on expiry | accept | Cache already serializes via single module-scope promise per Node process; acceptable load characteristics per Plan 04-01 operational evidence |
| T-05-01-08 | Repudiation | Sub-agent create/deactivate has no audit trail | mitigate | Route handler logs structured event `AGT-CREATE agency={agency_id} created_user={userId} role={role} by={caller_sub}` / `AGT-DEACTIVATE ... ` to Serilog; Keycloak itself records user events; together they provide two-layer audit |

      jwt: async ({ token, account, profile }) => {
        if (account && profile) {
          // Keycloak emits realm_access.roles; read and cache on the JWT
          // @ts-expect-error -- profile type doesn't know about realm_access

          token.sub = profile.sub ?? token.sub;
        }
        return token;
      },
      session: async ({ session, token }) => {
        if (token) {
          session.user.id = (token.sub as string) ?? session.user.id;

          (session as any).roles = (token.roles as string[] | undefined) ?? [];

      },
    },
