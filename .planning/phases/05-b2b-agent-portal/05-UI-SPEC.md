## Spacing Scale

Declared values (Tailwind v4 default 4px grid ‚Äî multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, inline badge padding, TTL chip internal padding |
| sm | 8px | Form field internal padding, chip/tag gaps, compact stack gap, dual-pricing table cell padding |
| md | 16px | Default element spacing, card internal padding, vertical rhythm between form fields |
| lg | 24px | Section padding inside cards, gap between dashboard cards, gap between result rows on comfortable-density pages |
| xl | 32px | Layout column gaps, page section breaks |
| 2xl | 48px | Major section breaks (dashboard header vs content, admin header vs sub-agent table) |
| 3xl | 64px | Page-level top/bottom padding on login, top-up confirmation |

**Phase 5 additional constants (not new tokens ‚Äî fixed layout values to prevent drift):**
- **Compact table row height: `h-11` = 44px** on agent tables (booking list, sub-agent list, wallet transactions, TTL alerts). Comfortable Phase 4 cards do not apply to agent data tables ‚Äî see "Information Density Contract" below.
- **Wallet chip height: `h-8` = 32px** in the header bar. Internal padding `px-sm py-xs` (8px / 4px). Icon 16px, amount 14px label weight.
- **TTL chip height: `h-6` = 24px** in table rows and dashboard cards. Internal padding `px-xs` (4px). Icon 12px, text 12px.
- **44px minimum touch target** on all interactive controls on mobile (inherited from Phase 4).
- **Header shell height: `h-14` = 56px desktop / `h-16` = 64px mobile** ‚Äî taller on mobile so the wallet chip remains readable without shrinking.

Exceptions: none beyond the constants above.

## Typography

Sizes locked at 4 roles (identical to Phase 4). Heading line-height 1.2, body line-height 1.5.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 600 (semibold) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.3 |
| Display | 28px | 600 (semibold) | 1.2 |

Rules:
- Only **two weights** site-wide: 400 (regular) and 600 (semibold). No 500, 700, 800 usage.
- **All money amounts use `font-variant-numeric: tabular-nums`** ‚Äî mandatory on the dual-pricing table, wallet balance, wallet transactions, invoice preview, TTL countdowns. This is tighter than Phase 4 (where it was recommended); in Phase 5 it is a checker-validated rule because numerical column alignment in the 4-column NET/markup/GROSS/commission grid depends on it.
- Prices and balances render at body size (16px) + semibold, never display size.
- 12px/400/1.4 reserved exclusively for disclaimer/fine print (invoice footer notes, fare rules small print, TTL chip label text).
- Agent-admin‚Äìonly surfaces (admin/, wallet top-up CTA) display a **`data-admin-only` label at 12px semibold** below the nav item using the muted-foreground token. Never a colored label.

## Color

Base palette inherited from Phase 4 (`ui/starterKit/styles/globals.css` ‚Äî zinc scale + white + red-600 destructive). Phase 5 introduces exactly **one** B2B-specific accent.

### Portal Differentiation Strategy

**Decision (defensible default, [ASSUMED ‚Äî revisit in /gsd-discuss-phase 5]):** Distinct accent color for B2B (`indigo-600`) **+** a persistent "AGENT PORTAL" wordmark badge in the header shell. Both-together chosen because:
1. Accent alone can be missed by colorblind users and on grayscale screenshots.
2. Badge alone is easy to glance past once familiarity builds.
3. Together they are redundant ‚Äî a safety property (if a cookie ever leaks and renders a B2C page inside b2b-web, the mismatch is instantly visible).

Indigo chosen over other hues because:
- It is visually distinct from B2C's `blue-600` (indigo is perceptibly more violet under sRGB; Œî-hue ‚âà 18¬∞).
- Passes WCAG AA on both the light `--background` (contrast ratio ‚â• 4.6:1 for `indigo-600` on `white`) and dark `--background` (contrast ratio ‚â• 5.1:1 for `indigo-400` on `zinc-950`).
- Does not clash with amber/red status indicators used by TTL chips.
- Already present in the Tailwind v4 default palette ‚Äî no new palette definition needed.

### Palette

| Role | Value (light) | Value (dark) | Usage |
|------|---------------|--------------|-------|
| Dominant (60%) | `--background` = `white` | `zinc-950` | Page background, content surfaces, form field backgrounds |
| Secondary (30%) | `--secondary` / `--muted` = `zinc-100` | `zinc-800..900` | Cards, sidebar, nav, dashboard panels, data-grid row hover, wallet chip background (`zinc-100` light / `zinc-800` dark) |
| Accent (10%) | **`indigo-600`** | **`indigo-400`** (for text on dark) / **`indigo-600`** (for fills) | Reserved-for list below |
| Destructive | `red-600` | `red-600` | Destructive actions + insufficient-funds blocking panel + urgent TTL chip |
| Warning | `amber-600` | `amber-500` | 24h TTL warning chip only |
| Success | `green-600` | `green-500` | Confirmed / committed / top-up success dot indicators only |

### Accent (`indigo-600`) reserved-for list

Accent is applied **only** to the following elements. Any other use is a checker violation.

1. **Primary CTA button fill** ‚Äî `Search flights`, `Confirm booking (debit wallet)`, `Create sub-agent`, `Top up wallet`, `Download invoice`
2. **Selected-state indicator on flight/hotel/car result rows** in the dual-pricing grid (left 4px `indigo-600` border + 1px ring)
3. **Active nav item underline** in the agent portal header
4. **Focus ring** on keyboard-focused interactive elements (via `--ring` ‚Äî overridden to `indigo-500` in `b2b-web/styles/globals.css` delta; `zinc-400` default stays for secondary-focus like inputs that haven't taken action yet)
5. **"AGENT PORTAL" wordmark badge** in the header shell (outline style ‚Äî 1px `indigo-600` border + `indigo-600` text on `--background`, never filled)
6. **Wallet chip border** in the header (1px `indigo-600` border; balance text stays `--foreground`; never a filled indigo chip ‚Äî that would fight the CTA)
7. **Data-grid column header hover** for sortable columns in booking/wallet tables (subtle `indigo-600` text on hover, not a fill)

**Not allowed for accent:** body links (use `--foreground` + underline on hover), headings, card backgrounds, decorative icons, status dots in lists (use green/amber/red semantic tokens), the markup/commission cell fills in the dual-pricing grid (must be neutral ‚Äî accent would visually confuse "agent profit" with "clickable CTA").

### Destructive (`red-600`) reserved-for list

1. **Insufficient-funds blocking panel** ‚Äî full-width banner on `/checkout/confirm` when wallet balance < booking total. Background tint `red-50` light / `red-950/40` dark. Border 1px `red-600`. Icon `lucide:alert-octagon`. See copywriting + layout contract below.
2. **Urgent TTL chip** (`< 2h to deadline`) ‚Äî background `red-50` light / `red-950/40` dark, border `red-600`, text `red-700` light / `red-300` dark, icon `lucide:alarm-clock` 12px.
3. **Deactivate sub-agent confirmation button** in the destructive confirmation dialog.
4. **Void booking** button in booking detail (agent-admin only).
5. **Form validation error text and field border** (inherited from Phase 4).
6. **"Cancelled" booking status badge** in booking list.
7. **"Release" transaction row tint** in wallet transactions list (subtle `red-50` / `red-950/40` row background ‚Äî signals the transaction reduced reservation, distinct from a commit or top-up).

**Not allowed for destructive:** generic warnings (use `amber-600`), info messages, tooltips, low-balance wallet banner (that uses `amber-600` ‚Äî see warning row below).

### Warning (`amber-600`) reserved-for list

1. **24h TTL warning chip** ‚Äî background `amber-50` light / `amber-950/40` dark, border `amber-600`, text `amber-700` light / `amber-300` dark, icon `lucide:clock-alert` 12px.
2. **Low-balance wallet banner** ‚Äî appears sitewide at the top of every agent route when wallet balance < configurable threshold. Icon `lucide:wallet-minimal`. Dismissible per session via X button; reappears on next login or refresh.
3. **"Pending" booking status badge** (awaiting ticketing) in booking list.

### Success (`green-600`) reserved-for list

1. "Confirmed" / "Ticketed" status dot in booking list (dot only ‚Äî no fill, no badge text).
2. Wallet top-up success toast (`sonner` with `type: "success"`).
3. "Wallet committed" row indicator in wallet transactions ‚Äî small green dot to the left of the transaction row when that transaction is a commit.

### Dual-pricing grid cell rules (critical ‚Äî Pitfall 23)

The dual-pricing grid renders 4 numeric columns simultaneously. Color discipline prevents confusion between "clickable" and "profit":

- **NET** column ‚Äî `--foreground` text on `--muted` background tint (`zinc-100` light / `zinc-800` dark). Never accent or destructive.
- **Markup** column ‚Äî `--foreground` text on `--background` (no tint). Column header label "Agency markup" in `--muted-foreground`.
- **GROSS** column ‚Äî `--foreground` text on `--background`, **semibold** (it's the price the agent quotes the customer, so it must visually lead).
- **Commission** column ‚Äî `green-700` light / `green-300` dark text on `--background`. This is the **only** place green is used as a text color ‚Äî it signals "what the agent earns", which is a long-standing travel-industry convention. This is not a reassignment of the success token ‚Äî it's a domain-specific text color pinned exclusively to this column.

### Dark mode

Carried over from Phase 4 (defensible default, [ASSUMED ‚Äî revisit in /gsd-discuss-phase 5]). Reasoning: the starter-kit fork brings the `.dark` block for free; omitting dark mode in b2b-web would require deleting it from every forked component stylesheet (costly, brittle). Agents work long hours and appreciate dark mode. All Phase 5 accent/warning/success tokens above have explicit light+dark values.

### Status color map (booking list + dashboard)

| Status | Indicator |
|--------|-----------|
| Confirmed / Ticketed | `green-600` dot only (no badge fill) |
| Pending / Awaiting ticket | `amber-600` dot + label `Pending` |
| TTL warning (24h) | amber chip (see Warning reserved-for list) |
| TTL urgent (2h) | red chip (see Destructive reserved-for list) |
| Cancelled / Failed | `red-600` dot + label `Cancelled` / `Failed` |
| Wallet-reserved (mid-saga) | `zinc-500` dot + label `Reserving` |

## Copywriting Contract

Voice: **direct, transactional, second person ("you", "your agency"), present tense.** British English spelling (`customise`, `organisation`, `whilst`). **Tone is slightly more formal than Phase 4** (defensible default, [ASSUMED ‚Äî revisit in /gsd-discuss-phase 5]) ‚Äî agents are professionals, not casual shoppers. No exclamation marks. No emoji. Sentence case for all labels, titles, and buttons. Never "Submit".

Rationale for stricter tone: agent empty states and error messages must be **actionable** (tell the agent what to do next), not empathetic. An agent hitting insufficient funds needs "Top up wallet" ‚Äî not "Oops! Looks like you're a bit short on funds, no worries!".

## Screen-by-Screen Contract

Each screen below defines: purpose, data fields, layout, interaction rules, accessibility requirements. The planner translates each row into `<acceptance_criteria>` fields. The executor implements; the checker and auditor validate.

### 1. Login (unauthenticated)

- **Purpose:** kick off OIDC Authorization-Code flow against Keycloak `tbe-b2b` realm.
- **Layout:** single centred card, 384px wide, with agency logo (agency logo rendering is v2 ‚Äî v1 shows static Metronic brand), "AGENT PORTAL" wordmark badge, and a `Sign in with Keycloak` primary CTA. No local email/password field ‚Äî this is a redirect-only page.
- **Copy:** Heading `Agent portal`. Body `Sign in with your agency credentials.` CTA `Sign in`. Below card footer: `Bookings are governed by your agency's wallet balance and markup rules.`
- **Interaction:** clicking `Sign in` redirects to Keycloak. Auth.js callback brings the user back to `/dashboard` (or the original intended route via `callbackUrl` param).
- **Accessibility:** primary CTA focusable; header landmarks announced.

### 2. Header Shell (every authenticated route)

- **Purpose:** persistent navigation + wallet balance + identity context.
- **Layout (desktop):** single 56px bar. Left to right: brand wordmark `TBE` + "AGENT PORTAL" outline badge, nav links (`Dashboard`, `Search`, `Bookings`, `Admin` (agent-admin only)), right cluster (wallet chip, notifications bell, user menu with agency name + agent name + Sign out).
- **Layout (mobile):** 64px bar. Brand + badge collapse into a `lucide:menu` hamburger that opens a `sheet` (right side). Wallet chip stays visible in the bar at all breakpoints ‚Äî compresses currency code ("¬£ 125") rather than hiding (agents must see their balance at all times, B2B-06).
- **Wallet chip content:** `lucide:wallet` icon + formatted balance (e.g., `¬£ 1,250.00`) + optional low-balance indicator dot (amber). Border `indigo-600`, semibold tabular-nums, 14px.
- **Data:** wallet balance from `GET /api/wallets/{walletId}/balance`, TanStack Query `refetchInterval: 30_000`, `staleTime: 25_000` (research D-27). First render is **server-rendered by the RSC layout** ‚Äî the chip must not flicker from blank to filled on initial page load.
- **Interaction:** clicking the wallet chip navigates to `/admin/wallet` for admin; for non-admin agents it opens a `popover` showing the balance + last 3 transactions (read-only) + disabled `Top up` button with tooltip `Only your agency admin can top up the wallet.`
- **Accessibility:** wallet chip `aria-label="Wallet balance: ¬£1,250.00"`. "AGENT PORTAL" badge has `aria-label="Agent portal"` because the word "agent" is not evident to assistive tech from the indigo outline alone. Nav landmarks use `<nav>` with `aria-label="Primary"`.

### 3. Dashboard (`/dashboard`)

- **Purpose:** single-glance overview + TTL alerts.
- **Layout decision (defensible default, [ASSUMED ‚Äî revisit in /gsd-discuss-phase 5]):** **2-column grid on desktop (‚â• `lg`)**, single-column stack on mobile.
  - Top row (full-width on mobile): Ticketing-deadline alerts card ‚Äî amber and red TTL chips for bookings < 24h and < 2h. If empty, render the empty-state copy (no card chrome wasted on "nothing to do").
  - Row 2 left: Wallet balance card (balance, threshold indicator, last 5 transactions, "Top up" CTA for admin).
  - Row 2 right: Recent bookings card (5 most recent bookings with status dot + PNR + customer name + quick "View" link).
  - Row 3 full-width: Quick links grid ‚Äî 4 tiles (Search flights, Search hotels, Search cars, New sub-agent ‚Äî agent-admin only). 4-column on desktop ‚â• `lg`, 2-column on `md`, single-column below.
- **Data:**
  - `GET /api/b2b/bookings/me?urgent=true` ‚Üí TTL chips (cached 60s per research D-28).
  - `GET /api/wallets/{walletId}/balance` ‚Üí wallet card.
  - `GET /api/b2b/bookings/me?page=1&pageSize=5` ‚Üí recent bookings.
- **Interaction:** each TTL chip is a link to the booking detail. Tiles in the Quick Links grid keyboard-focusable and ‚â• 44px hit area.
- **Accessibility:** each card uses `<section aria-labelledby="...">`. TTL urgent chips have `role="alert"` with `aria-live="polite"` only on initial render (not on every poll refresh ‚Äî otherwise screen readers announce the same chip every 60s).

### 4. Search (`/search/flights`, `/search/hotels`, `/search/cars`)

- **Purpose:** identical to B2C search forms ‚Äî planner reuses the exact starterKit components and form schemas from `src/portals/b2c-web/components/search/`.
- **Delta from Phase 4:** none for the form itself. Difference appears in the results page.
- **Interaction:** same URL-state pattern via nuqs (D-11). Same airport/destination typeahead endpoints (OpenFlights via Redis for IATA, D-18).
- **Accessibility:** same as Phase 4.

### 5. Dual-Pricing Results Table (`/search/flights` results panel, analogous for hotel/car)

- **Purpose:** show agent all four price components simultaneously (B2B-03/04).
- **Layout (desktop):** stacked rows in a `data-grid-table` with columns:

## Interaction & Layout Contracts (phase-specific)

These are Phase 5 mechanics the planner and executor must follow verbatim. The checker validates against them.

### Portal Differentiation

- **The "AGENT PORTAL" outline badge** is rendered in the header shell on every authenticated route. `<AgentPortalBadge />` ‚Äî 1px `indigo-600` border, `indigo-600` text, `--background` fill, 12px semibold, uppercase letters, tracking-wide. Never filled; never below 32px total height.
- **Primary accent token** is `indigo-600` (light) / `indigo-400` text / `indigo-600` fills (dark), replacing Phase 4's `blue-600`. This is the **only** Tailwind palette swap between b2c-web and b2b-web.

### Wallet Chip & Balance Polling

- Wallet chip is server-rendered on initial page load (`layout.tsx` RSC calls `auth()` ‚Üí `gatewayFetch()` ‚Üí injects balance into the `<Header />` component). Client then takes over with TanStack Query `refetchInterval: 30_000, staleTime: 25_000`.
- Balance always read from the append-only ledger via the projection ‚Äî never from a cached balance column (Pitfall 21 inherited).
- Currency code is **GBP** in v1. If the agency wallet is in a different currency, the projection returns the correct currency; the chip renders whatever comes back ‚Äî no currency conversion in the UI.

### Dual-Pricing Grid

- Grid renders **all four columns simultaneously** on desktop and mobile. Never behind tabs, never on hover. Hard checker validation.
- Markup is **server-computed** exactly once per search result. Client reads `quote.gross` / `quote.markup` / `quote.commission` verbatim. No `import`s of markup config under `src/portals/b2b-web/` (Pitfall 23).
- Commission column uses `green-700` (light) / `green-300` (dark) text. **This is not the success token** ‚Äî it's a domain-specific text color allowed only in this column.
- Sort by commission is allowed (B2B-specific feature); sort by markup is allowed; sort by NET is allowed; sort by GROSS is default.

### Booking-on-Behalf

- Customer contact form persists `CustomerName/Email/Phone` on `BookingSagaState` (research Pattern 5).
- `Channel = B2B` stamp is server-side; client never sets it.
- Per-booking markup override is **agent-admin only** ‚Äî visible and editable only when session role includes `agent-admin`. Server enforces via `[Authorize(Policy = "B2BAdminPolicy")]` (research Pattern 2).

### Checkout Debit (No Stripe)

- `/checkout/confirm` contains **no `<Elements>` mount and no PaymentElement**. CSP for that route does NOT whitelist `js.stripe.com` ‚Äî only `/admin/wallet/*` does.
- The Confirm CTA triggers `POST /api/b2b/bookings` which initiates the saga. Saga branch publishes `WalletReserveCommand` on `PnrCreated` (research Pattern 4 / Example 1).
- Success URL has `?booking={id}`, never `?payment_intent=...` (Pitfall 6 inherited).

### Insufficient-Funds State

- Detection: server returns `409 InsufficientFunds` from `POST /api/b2b/bookings` (or the wallet reservation fails via `WalletReservationFailed` event). Client reads both paths and renders the same panel.
- Panel **replaces** the Debit summary card in-place. Confirm CTA is **removed from the DOM**, not just disabled.
- Non-admin agents see the non-admin variant copy and the `Request top-up from your admin` CTA (mailto).
- Admin agents see the admin variant copy and the `Top up wallet` CTA (`/admin/wallet`).
- Banner **cannot** be dismissed ‚Äî it persists until the user either tops up or changes the booking.

### TTL Chips

- Data source: booking projection returns `TicketingDeadlineUtc` (UTC ISO string). Client computes the time-remaining in the render and classifies into `normal` (> 24h ‚Äî no chip), `warning` (24h‚Äì2h ‚Äî amber chip), `urgent` (< 2h ‚Äî red chip).
- Rendering: chip includes an icon (`lucide:clock-alert` for warning, `lucide:alarm-clock` for urgent) + text ("24h to ticket" / "Urgent: 1h 23m left"). **Not color-only** ‚Äî the icon + text ensure colorblind accessibility (WCAG 1.4.1).
- Chips appear in: booking list `TTL` column, dashboard TTL alerts card, booking detail Status card.
- Countdown updates client-side at 1Hz; data refetch at 60s via TanStack Query.

### Sub-Agent Create Flow

- Dialog form collects `firstName`, `lastName`, `email`, `role` (radio).
- On submit, client calls `POST /api/b2b/agents`. Server-side handler calls Keycloak Admin API via `lib/keycloak-b2b-admin.ts` using the `tbe-b2b-admin` service-account token (research Pattern 6).
- **Server always stamps `agency_id` from the caller's session** (Pitfall 28). The client never sees the `agency_id` value ‚Äî it's never in request body or URL.
- New user gets `emailVerified: false` and Keycloak sends a verification email automatically (or manually triggered depending on realm settings).

### Forms, Loading, and Feedback

Inherited from Phase 4 verbatim:
- `react-hook-form` + `zod`. Validation errors below each field in `red-600` 14px; red 1px field border.
- Skeleton rows matching final card/row shape on results/list pages.
- `sonner` toasts for non-blocking successes only. Errors that require user action use inline alerts.
- Focus rings using the `--ring` token. **Phase 5 override:** `--ring` set to `indigo-500` (light) / `indigo-400` (dark) so focus outlines match the accent.

### Responsive Breakpoints

Follow Tailwind v4 defaults: `sm` 640 / `md` 768 / `lg` 1024 / `xl` 1280 / `2xl` 1536. Mobile-first.
- Header shell: hamburger at `md` and below.
- Dashboard: 2-column grid activates at `lg`.
- Dual-pricing grid: table at `lg`+, stacked cards below.
- Booking list: table at `md`+, stacked row-cards below.
- Wallet top-up: single-column everywhere (narrow form).

### Accessibility Baseline (WCAG 2.1 AA)

- All interactive elements keyboard-reachable; tab order matches visual order.
- All form fields have explicit `<label>` (not placeholder-only).
- Dialogs use Radix `Dialog` / `AlertDialog` ‚Äî focus trapped, ESC closes, focus returns to trigger.
- All text meets 4.5:1 contrast on its background; large text (18.66px+) meets 3:1.
- **Color is never the only channel**: TTL chips have icon + text; status indicators have dot + label (except the Confirmed status which uses dot-only ‚Äî but "Confirmed" is the implicit default for any row without another badge, and the dashboard legend documents the color map).
- **"AGENT PORTAL" badge is announced** via `aria-label="Agent portal"` ‚Äî the word "agent" is not evident from the visual alone to screen readers.
- Wallet chip announces as `"Wallet balance: ¬£1,250.00"` via `aria-label`.
- TTL countdowns use `aria-live="polite"` only on threshold transitions (not on every tick).
- Insufficient-funds panel uses `role="alert"` on mount.
- Low-balance banner uses `role="status"` (not `alert` ‚Äî it's informational).
