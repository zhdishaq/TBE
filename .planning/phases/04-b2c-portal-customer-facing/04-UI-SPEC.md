# Phase 4 — UI Design Contract

> Visual and interaction contract for the B2C portal. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Pre-populated from `04-CONTEXT.md` decisions D-01..D-19 and the Metronic 9 starterKit baseline at `ui/starterKit/`.

| Property | Value |
|----------|-------|

## Spacing Scale

Declared values (Tailwind v4 default 4px grid — must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px  | Icon-to-label gaps, inline badge padding |
| sm | 8px  | Form field internal padding, chip/tag gaps, compact stack gap |
| md | 16px | Default element spacing, card internal padding, vertical rhythm between form fields |
| lg | 24px | Section padding inside cards, gap between result cards in results list |
| xl | 32px | Layout column gaps (Trip Builder flight/hotel split), page section breaks |
| 2xl | 48px | Major section breaks (hero vs search form, dashboard header vs content) |
| 3xl | 64px | Page-level top/bottom padding on marketing surfaces (landing, login, confirmation) |

Exceptions:
- **44px minimum touch target** on all interactive controls on mobile (buttons, segmented passenger counters, date cells). Not a spacing token — a minimum hit area requirement enforced via `min-h-11` / `min-w-11`.
- **90px sticky basket footer height** on mobile Trip Builder (not a token — a fixed layout value documented here to prevent drift).

## Typography

Sizes locked at 4 roles. Heading line-height 1.2, body line-height 1.5.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 600 (semibold) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.3 |
| Display | 28px | 600 (semibold) | 1.2 |

Rules:
- Only **two weights** used site-wide: 400 (regular) for body/paragraph text, 600 (semibold) for labels, headings, display, and CTA button labels. No 500, 700, 800 usage.
- **Prices are body size + semibold** — never a larger display size; prices must not compete with headings.
- Small disclaimers (fare rules fine print, "taxes included") render at **12px / 400 / 1.4** — this is the only approved size outside the four roles, reserved for legal/disclaimer text only.
- Tabular numerals (`font-variant-numeric: tabular-nums`) required for all money amounts and passenger counts so column alignment stays stable.

## Color

Base palette inherited from `ui/starterKit/styles/globals.css` (zinc scale + white + red-600 destructive). Accent is the **only** non-neutral non-destructive hue in Phase 4.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` = white (light) / `zinc-950` (dark) | Page background, content surfaces, form field backgrounds |
| Secondary (30%) | `--secondary` / `--muted` = `zinc-100` (light) / `zinc-800..900` (dark) | Cards, sidebar, nav, search-form container, result card chrome, passenger counter backgrounds, dashboard panels |
| Accent (10%) | **`blue-600`** (`oklch` equivalent from Tailwind v4 `--color-blue-600`) | Reserved-for list below — do not apply anywhere else |
| Destructive | `--destructive` = `red-600` | Destructive actions only — see reserved-for list below |

**Accent (`blue-600`) reserved for:**
1. Primary CTA button fill (Search, Continue to payment, Pay £X, Download receipt)
2. Selected-state indicator on flight/hotel result cards (left border + subtle ring)
3. Selected-state on passenger-counter segmented controls and cabin-class pills
4. Active nav item underline in dashboard header
5. Focus ring on keyboard-focused interactive elements (via `--ring`)
6. Trip Builder basket "component added" check icon

Not allowed for accent: body links (use `foreground` + underline on hover), headings, card backgrounds, icons that are purely decorative, status badges other than "selected".

**Destructive (`red-600`) reserved for:**
1. Cancel booking confirmation button
2. Remove-from-basket action in Trip Builder
3. Payment failure / hotel-unavailable banner background tint
4. Form validation error text and field border
5. "Cancelled" booking status badge in dashboard

Not allowed: generic warnings (use `amber-600` via chart-3 only for TTL/ticketing-deadline advisory indicators on the dashboard), info messages, tooltips.

**Status color map (dashboard + trip list):**
- Confirmed / Ticketed → `green-600` dot only (no fill)
- Pending / Awaiting payment → `zinc-500` dot only
- TTL warning (< 24h) → `amber-600` dot + label
- Cancelled / Failed → `red-600` dot + label

Dark mode: all mappings from `globals.css` `.dark` block apply unchanged. Accent `blue-600` stays identical in both themes.

## Copywriting Contract

Voice: direct, factual, second person ("you", "your booking"), present tense. British English spelling throughout (`customise`, `organisation`, `whilst`) to match the British travel market. No exclamation marks. No emoji in product UI. Sentence case for all labels, titles, and buttons — **not** Title Case.

### Global CTAs

| Element | Copy |
|---------|------|
| Primary CTA (search) | `Search flights` / `Search hotels` / `Search cars` (verb + noun — never "Submit" or "Go") |
| Primary CTA (trip builder basket empty) | `Build your trip` |
| Primary CTA (checkout entry) | `Continue to payment` |
| Primary CTA (payment step) | `Pay £{amount}` (render the live basket total in the label) |
| Primary CTA (booking success) | `Download receipt` |
| Secondary CTA (results filter reset) | `Clear filters` |
| Secondary CTA (results back) | `Change search` |

### Empty states

| Surface | Heading | Body |
|---------|---------|------|
| Dashboard — no upcoming trips | `No upcoming trips` | `When you book a flight, hotel, or car, it will appear here. Search now to get started.` + `Start a search` link |
| Dashboard — no past bookings | `No past bookings yet` | `Your booking history will show here once you have completed a trip.` |
| Search results — zero flights | `No flights found for these dates` | `Try adjusting your dates, cabin class, or nearby airports.` + `Change search` button |
| Search results — zero hotels | `No hotels available for these dates` | `Try different dates or a wider destination area.` + `Change search` button |
| Trip Builder — basket empty | `Your trip is empty` | `Pick a flight and a hotel to build your trip. Each item keeps its own booking reference and cancellation policy.` |
| Airport autocomplete — no match | `No airports match "{query}"` | `Try a city name or 3-letter airport code (e.g. LHR, JFK).` |

### Error states

| Surface | Copy |
|---------|------|
| Search failed (gateway / GDS timeout) | `We couldn't reach our flight partners just now. Please try again in a moment.` + `Retry` button |
| Price changed at review step | `The price for this flight has changed since you selected it. New total: £{new_total}. Continue or go back to results.` |
| Email not verified (at checkout gate) | `Please verify your email before booking. We sent a link to {email}.` + `Resend verification email` link |
| Payment card declined | `Your card was declined. Please try another card or contact your bank.` |
| Partial basket failure (flight OK, hotel gone) | `Your flight is confirmed: {PNR}. The hotel you picked is no longer available, so only your flight was charged. Choose another hotel?` + `Find another hotel` button |
| Partial basket failure (hotel OK, flight gone) | `Your hotel is confirmed: {supplier_ref}. The flight you picked sold out, so only your hotel was charged. Find another flight?` + `Find another flight` button |
| Session expired | `Your session timed out. Please sign in to continue.` + `Sign in` button |
| Wallet/auth generic | `Something went wrong on our end. Please try again. If the problem continues, contact support.` |

### Destructive confirmations

Every destructive action requires an explicit confirmation dialog with a two-button footer: a neutral `Keep booking` / `Keep in basket` on the left, and the destructive `red-600` button on the right. The dialog title must name the action in the imperative.

| Action | Dialog title | Dialog body | Confirm button |
|--------|--------------|-------------|----------------|
| Cancel a confirmed booking | `Cancel this booking?` | `Cancellation is subject to the fare rules for this booking. Refund eligibility: {rule_summary}. This cannot be undone from the portal.` | `Cancel booking` |
| Remove flight from Trip Builder basket | `Remove flight from your trip?` | `The hotel you picked will stay in your basket. The price may change if you add a different flight later.` | `Remove flight` |
| Remove hotel from Trip Builder basket | `Remove hotel from your trip?` | `The flight you picked will stay in your basket. The price may change if you add a different hotel later.` | `Remove hotel` |
| Abandon checkout (back nav during payment) | `Leave checkout?` | `Your basket will be kept for 10 minutes. Prices may change if you come back later.` | `Leave checkout` |

### Success messages

| Surface | Copy |
|---------|------|
| Booking confirmed (flight only) | `Flight booked. Booking reference: {PNR}. Your e-ticket will arrive by email within 60 seconds.` |
| Booking confirmed (hotel only) | `Hotel booked. Booking reference: {supplier_ref}. Your voucher will arrive by email within 60 seconds.` |
| Trip booked (flight + hotel, both succeeded) | `Trip booked. Flight: {PNR}. Hotel: {supplier_ref}. Your confirmation will arrive by email within 60 seconds.` |
| Email verification sent | `Verification email sent to {email}. It may take a minute to arrive.` |
| Password reset sent | `Password reset link sent. Check your email.` |

## Interaction & Layout Contracts (phase-specific)

These are Phase 4 mechanics the planner and executor must follow. The checker validates against them.

### Search form
- Layout: single horizontal row on desktop ≥ `lg` (≥1024px), stacked vertical on smaller. Fields in order: trip type (one-way / return / multi-city) → origin → destination → depart → return → passengers → cabin class → `Search flights` button.
- Origin/destination: combobox with IATA autocomplete (D-18). Show "LHR — London Heathrow" pattern (code em-dash name). Minimum 2 chars before querying.
- Date picker: `react-day-picker` (D-19). Return trip shows a range; minimum selectable date is today; maximum 361 days out.
- Passengers: popover with four counters (adult, child 2-11, infant on lap <2, infant in seat <2). Max total 9. Validation: infants on lap ≤ adults; display inline help text, not a toast.

### Results page
- URL state: all search params + filters + sort encoded via `nuqs` or equivalent (D-11). Browser back/forward must restore state without re-querying unless search criteria change.
- Layout: left-hand **filter rail** (280px fixed on desktop, bottom sheet on mobile) + right-hand **stacked result cards** (no pagination, server caps at 200) per D-13.
- Filter rail: stops (radio: any / direct / 1 stop / 2+ stops), airline (checkbox list), departure time windows (4 chips: early morning / morning / afternoon / evening), price range (slider). Filters run client-side over the TanStack Query cache (D-12).
- Each flight card shows: airline logo (24px) + name, departure time + IATA, arrow, arrival time + IATA, total duration, stop count, baggage icon with allowance, all-in price right-aligned with small "incl. taxes" label. Tapping/clicking a card opens a details drawer with fare rules breakdown (FLTB-02) before "Select".
- Selected state: left 4px `blue-600` border + 1px ring. Single selection at a time per panel.

### Trip Builder
- Layout: side-by-side flight panel (left) + hotel panel (right) on desktop ≥ `lg`; tabbed on smaller screens (D-07). Shared sticky basket footer shows both selections and combined total.
- Basket footer (D-08): two stacked line items (flight + hotel) with individual refs and individual "Remove" actions; combined total line with VAT/YQ-YR breakdown disclosed in a details toggle; `Continue to payment` CTA.
- Partial-failure banner (D-09): full-width, `red-600` tint, icon + heading + body copy + one action button. Only one partial-failure state at a time.

### Checkout
- Maximum 5 visible steps on mobile (B2C-05 / D-14). Stepper at top shows: `Search → Results → Select → Details → Payment`. Confirmation is a separate post-checkout screen and does not count.
- Email verification gate (D-06) fires on entering step 5 (payment). Modal: heading `Verify your email first`, body `We need to confirm your email before you can pay. We sent a link to {email}.`, primary `Resend email`, secondary `I have verified, refresh`.
- Stripe Payment Element: full-width field, no surrounding card chrome (Stripe Elements provides its own). Submit button below shows live amount: `Pay £{amount}`.

### Dashboard
- RSC page reading from `GET /customers/me/bookings` (D-17). Two tabs: `Upcoming` (default, sorted by soonest departure) and `Past` (reverse chronological).
- Booking list row: product icon (plane / bed / car) + route/property + dates + status badge + reference + chevron. Tapping/clicking opens detail page with itinerary + `Download receipt` button wired to `GET /api/bookings/{id}/receipt.pdf` (D-15).
- Empty state copy from the Copywriting Contract above.

### Forms, loading, and feedback
- All forms use `react-hook-form` + `zod`. Validation errors appear below each field in `red-600` 14px text, and as a red 1px field border. No validation toasts.
- Loading: skeleton rows matching final card shape on results pages (no spinners in main content area). Suspense boundaries wrap RSC data reads. Inline button spinners allowed on CTAs while a mutation is in flight — button label stays visible.
- Toasts (`sonner`): success toasts for non-blocking confirmations only (e.g. "Email sent"). Errors that require user action use inline alerts, not toasts.
- Focus: all interactive elements must show a visible focus ring using `--ring`. Never remove `outline` without replacement.

### Responsive breakpoints
Follow Tailwind v4 defaults: `sm` 640 / `md` 768 / `lg` 1024 / `xl` 1280 / `2xl` 1536. Design targets mobile-first; desktop layouts are the enhanced path. Trip Builder side-by-side activates at `lg`.

### Accessibility baseline
- WCAG 2.1 AA contrast minimum for all text on `--background` and on accent / destructive surfaces.
- All interactive elements keyboard reachable; tab order matches visual order.
- Form fields have explicit `<label>` (not placeholder-only labels).
- Dialogs use Radix `Dialog` primitive — focus trapped, ESC closes, focus returns to trigger.
