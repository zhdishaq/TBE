<objective>
Ship car hire + transfer search surfaces, a car voucher email pipeline, and the full Trip Builder (flight + hotel in one basket with a combined checkout). This plan is the capstone of Phase 4 — it closes all remaining phase requirements (CARB-01..03, PKG-01..04) by composing primitives from 04-00/01/02/03.

**Committed Stripe strategy — Option A (two PaymentIntents):** Per RESEARCH Pitfall 9 open question, one PaymentIntent per leg (flight + hotel), each with `capture_method: "manual"`, each with a deterministic idempotency key (`basket-{basketId}-authorize-{leg}`). This aligns with D-08 ("BookingService creates two independent bookings with their own references"), Phase 3 D-13 idempotency-per-operation, and avoids Stripe's single-capture-per-PI constraint. Customer sees ONE Stripe Element UI that authorizes both in sequence; statement shows two separate charges (acceptable tradeoff — rationale copied into the basket-footer disclosure per UI-SPEC).

Purpose: Customers can book a combined flight+hotel in a single checkout with clean partial-failure behavior, and can book car hire / transfers.
Output: Full Phase 4 B2C portal feature set is shippable after this plan.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js server | Auth.js session cookie; basket state in sessionStorage (non-sensitive — only offer IDs, no prices trusted) |
| Next.js server → YARP gateway → BookingService | Bearer JWT injected server-side; request bodies carry Guest PII |
| BookingService → PaymentService/Stripe | Idempotency keys are deterministic server-derived; never from client |
| Stripe webhook → BookingService | Webhook signature verification (Phase 3 pattern, reused) |
| BasketPaymentOrchestrator → MassTransit/RabbitMQ | Internal bus; BasketConfirmed/BasketPartiallyConfirmed carry PII (email + guest name) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | T (Tampering) | Client-submitted basket totals | mitigate | Server re-prices every OfferId on POST /baskets; `totalAmount` from request body is ignored (server-computed) |
| T-04-04-02 | T | Stripe idempotency key collision | mitigate | Keys deterministically derived from basketId + operation + leg (`basket-{id}-authorize-flight`); never client-generated (Phase 3 D-13) |
| T-04-04-03 | I (Info Disclosure) | Basket GET cross-user | mitigate | Ownership check in BasketsController GET — `basket.UserId != userId && !IsInRole("backoffice-staff") → Forbid()` |
| T-04-04-04 | R (Repudiation) | Duplicate capture on webhook replay | mitigate | BasketPaymentOrchestrator checks a per-basket event log before each CaptureAsync; Stripe idempotency key also guards server-side |
| T-04-04-05 | D (DoS) | Basket spam creating orphaned PIs | mitigate | Rate-limit POST /baskets at 10/min per userId; orphaned baskets expire via a background cleanup job (out of scope for this phase — tracked as retro item) |
| T-04-04-06 | E (Elevation) | Partial-failure email leaking other customer's data | mitigate | EmailIdempotencyLog keyed by (EventId, EmailType); BasketConfirmationModel built exclusively from the event payload — no cross-customer DB reads |
| T-04-04-07 | S (Spoofing) | Forged BasketConfirmed event | mitigate | Event only publishable by BasketPaymentOrchestrator within BookingService; RabbitMQ credentials kept server-side (Phase 1 topology) |
| T-04-04-08 | T | Migration drift (Basket + CarBooking tables) | mitigate | Hand-authored migrations with deterministic timestamps and `Down()` methods |
| T-04-04-09 | I | Car booking voucher PDF disclosure | mitigate | Same ownership check + streaming pattern as hotel voucher (04-03) |
| T-04-04-10 | R | D-09 partial-failure refund dispute | mitigate | BasketPartiallyConfirmed event records `ChargedAmount` + `RefundedAmount`; email includes both; PaymentService ledger persists every VoidAsync call with idempotency key |
